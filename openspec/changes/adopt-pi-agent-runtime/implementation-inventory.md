# Pi runtime migration implementation inventory

Date: 2026-07-17

This inventory freezes the retained callers and ownership decisions before the Pi replacement begins. Paths describe the current source tree; tests under the same path follow the production owner unless a row explicitly says they are replacement characterization tests.

## Classification

| Classification | Meaning                                                                                                                                |
| -------------- | -------------------------------------------------------------------------------------------------------------------------------------- |
| Pi-owned       | Replace the OpenNeko implementation with the public Pi primitive and remove the old implementation.                                    |
| OpenNeko-owned | Retain the product responsibility and its authoritative data, but remove dependencies on the old Agent kernel.                         |
| Moved          | Retain the behavior under a narrower owning domain or Host boundary; do not preserve the current abstraction merely for compatibility. |
| Deleted        | Remove the implementation, exports, tests, configuration aliases, and successful call path after consumers use Pi.                     |

## Sequencing constraint

The replacement boundary is TUI + VS Code Extension + Webview together. During implementation, deterministic tests may instantiate the new Pi composition directly, but no product build may expose a runtime flag, dual registration, fallback, or compatibility alias that can return success through the old path. Removal occurs only after the retained Host and domain consumers above use the canonical path.

The VS Code Pi runtime manager owns one program-level `NodePiConversationAuthority` rather than opening one SQLite owner per conversation. Catalog create/list/title/delete operations share that authority; conversation owners borrow it and release only their fenced execution lease. `ConversationBridge` now lazily projects active Pi branch entries into Webview messages and uses Pi catalog metadata for listing, while Extension/TUI SQLite bindings are explicitly named local metadata and expose no transcript storage or recovery contract. Deletion removes authoritative SQLite catalog metadata and mapped Pi JSONL, and the asynchronous conversation-control chain waits for it before clearing the Webview projection. Legacy ConversationManager, Journal reader/projection/storage, resume hydration, transcript migration/import files, and AgentSession-owned custom compaction are deleted; `scripts/check-neko-agent-boundaries.mjs` rejects their source paths and production symbols in both Hosts. Task 4.4 is complete.

The generic AI SDK MediaAdapter compatibility bridge, its provider allowlist, the unused TUI Platform-chat perception pipeline, and the legacy AgentSession/Executor consumers are deleted. NewAPI-specific native media models and provider-owned media adapters remain, with deterministic tests proving that neither path can fall back to the other. Character dialogue/profile and embedding no longer use `Platform.createService()`: Character uses semantic ports backed by the exact Pi purposes, while embedding uses a dedicated exact-binding domain runtime for NewAPI/OpenAI-compatible and Ollama endpoints. Task 4.2 is complete.

The shared deletion step is an exact-purpose completion primitive that receives only one frozen model-use snapshot and bounded context. `image.understand` may consume it from the Agent tool bridge, while Canvas and Character consume it through owning-domain adapters assembled outside Agent. The helper does not select models or introduce a generic service facade. Both domains expose only semantic ports and know no provider/model/auth/Pi details; `text.embed` is isolated behind an exact frozen binding and does not reuse Platform chat/service adapters.
